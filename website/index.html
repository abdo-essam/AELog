Quick Setup

Add AELog to your project

Four steps to full in-app debugging. Pick only the plugins you need — every artifact is independently optional.

1

Add your dependency

Pick the plugins you want. Each artifact carries ae-log-core transitively — no need to add it manually.

build.gradle.kts
💡 Using Version Catalog?
libs.versions.toml
[versions]
aelog = "1.0.5"

[libraries]
aelog-logs           = { module = "io.github.abdo-essam:ae-log-logs",           version.ref = "aelog" }
aelog-network-ktor   = { module = "io.github.abdo-essam:ae-log-network-ktor",   version.ref = "aelog" }
aelog-network-okhttp = { module = "io.github.abdo-essam:ae-log-network-okhttp", version.ref = "aelog" }
aelog-analytics      = { module = "io.github.abdo-essam:ae-log-analytics",      version.ref = "aelog" }
aelog-crashes        = { module = "io.github.abdo-essam:ae-log-crashes",        version.ref = "aelog" }
2

Initialize AELog

Call this early in your platform entry point — Application.onCreate() for Android, or your KMP main entry.

Application.kt / main.kt
AELog.init(
    LogPlugin(),       // Logcat-style viewer
    NetworkPlugin(),   // HTTP request inspector
    AnalyticsPlugin(), // Event tracker
    CrashPlugin()      // Crash reporter — captures fatals automatically!
)
ℹ️ All calls are silent no-ops if init() hasn't been called — safe in shared modules.
3

Wrap your root Composable

Wrap your app once with LogProvider. This is the only UI change required. The enabled flag means zero overhead in release builds.

App.kt
@Composable
fun App(isDebug: Boolean) {
    LogProvider(
        enabled = isDebug, // ← zero overhead in release builds
        uiConfig = UiConfig(
            showFloatingButton = true,
            enableLongPress    = true,
        )
    ) {
        MaterialTheme {
            YourAppContent() // No other changes needed
        }
    }
}
Any Activity.kt
// No Compose required — launch the built-in Activity
import com.ae.log.launchViewer

// From a button click, developer menu, or shake gesture:
AELog.launchViewer(requireContext())
4

Start logging

Use the static AELog API anywhere in your codebase. Tap the bug button to open the inspector.

Anywhere in your app
// Logs — mirrors Android Log.*
AELog.log.v("Auth", "Checking token")
AELog.log.d("Auth", "Token refreshed")
AELog.log.e("DB", "Write failed", exception)

// Network — or use the Ktor/OkHttp interceptor for auto-capture
AELog.network.logRequest("GET", "https://api.example.com/users")

// Analytics events
AELog.analytics.logEvent("purchase", mapOf("item" to "premium", "price" to "9.99"))
AELog.analytics.logScreen("HomeScreen")

// Non-fatal crash capture
AELog.crashes.recordNonFatal(exception)
🎯 Auto-tag: Omit the tag and AELog derives it from the caller's class name — no repetition needed.

Built for Scale

A multi-layered architecture that guarantees zero-data-loss and minimal performance overhead on the host application.

Core Engine

AELog Core

The heartbeat of the system. Manages Plugin Registry, PluginContext lifecycle, UI rendering loop, and the global Singleton API.

// Initialize with any plugins
AELog.init(
  LogPlugin(),
  CrashPlugin(),
  NetworkPlugin(),
  AnalyticsPlugin()
)
⚡

Zero Overhead

The internal RingBuffer stores logs efficiently in memory, preventing out-of-memory errors and ensuring logging never blocks your main UI thread.

💾

Pluggable Storage

Store crashes and logs efficiently using file-based persistence that survives restarts.

🔌

Modular Plugins

Feature-agnostic core with independent modules for Network, Crashes, and Sinks.
